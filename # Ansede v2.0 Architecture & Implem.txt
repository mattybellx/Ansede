# Ansede v2.0: Architecture & Implementation Specification

## Executive Summary
Ansede is an ambitious, functional prototype that attempts to solve a domain-hard problem (Zero-Dependency SAST) by tightly coupling parsing, intermediate representation (IR), and rule evaluation entirely in Python. While this achieved fast initial velocity, it is structurally a technical debt trap. 

To reach v2.0 (Enterprise-Grade), Ansede must enforce a strict three-layer separation: **Parse $\rightarrow$ Represent $\rightarrow$ Evaluate**. This specification outlines the complete overhaul required to move away from hardcoded AST visits to a decoupled, highly performant, and scalable engine.

---

## Phase 1: Rule Engine Decoupling (The Python DSL)

Hardcoding `_rule_NN(ctx: _Ctx)` functions inside core analyzer files violates the Open-Closed Principle and forces O(N) AST visits.

### 1. Implement a Typed Rule Registry
Create a Python DSL using decorators so rules can subscribe to specific node types. This decoupling means adding a new rule never touches engine core code.

```python
from collections import defaultdict
from dataclasses import dataclass
from typing import Optional, List

class RuleRegistry:
    _rules = defaultdict(list)
    
    @classmethod
    def register(cls, *node_types):
        def decorator(rule_cls):
            for nt in node_types:
                cls._rules[nt].append(rule_cls())
            return rule_cls
        return decorator

@RuleRegistry.register('CALL', 'FORMATTED_STRING')
class SQLInjectionRule:
    cwe = "CWE-89"
    severity = "critical"
    
    def evaluate(self, node, ctx) -> Optional[dict]:
        # Taint evaluation logic goes here
